## 1. NSAllowsArbitraryLoadsInWebContent (iOS ATS)

### 설명

iOS의 **App Transport Security(ATS)** 는 앱의 네트워크 보안을 강제하는 기능이다. `NSAllowsArbitraryLoadsInWebContent: true`를 `Info.plist`에 추가하면 **WKWebView 내에서** HTTP/HTTPS 리소스 로딩을 허용한다. Capacitor 앱은 WKWebView 기반이므로 이 설정이 없으면 외부 리소스(카카오맵 타일 등)가 차단될 수 있다.

### 코드 예시

```xml
<!-- ios/App/App/Info.plist -->
<key>NSAppTransportSecurity</key>
<dict>
    <key>NSAllowsArbitraryLoadsInWebContent</key>
    <true/>
    <key>NSExceptionDomains</key>
    <dict>
        <key>daumcdn.net</key>
        <dict>
            <key>NSIncludesSubdomains</key>
            <true/>
            <key>NSExceptionAllowsInsecureHTTPLoads</key>
            <true/>
        </dict>
    </dict>
</dict>
```

### 핵심 포인트

- `NSAllowsArbitraryLoads` — 앱 전체 HTTP 허용 (Apple 심사에서 거부될 수 있음)
- `NSAllowsArbitraryLoadsInWebContent` — **WebView 내부만** 허용 (심사 통과 가능)
- `NSExceptionDomains` — 특정 도메인만 HTTP 허용 (가장 세밀한 제어)

### 참고

- [Apple ATS 공식 문서](https://developer.apple.com/documentation/bundleresources/information_property_list/nsapptransportsecurity)

## 2. React Query staleTime & invalidation으로 UI 갱신 시점 제어

### 설명

React Query의 `invalidateQueries`는 캐시를 무효화하고 즉시 refetch를 트리거한다. 찜 목록처럼 **해제해도 바로 사라지면 안 되는 UI**에서는, mutation의 `onSuccess`에서 해당 쿼리의 invalidation을 제거하고 `staleTime: 0`을 활용해 **페이지 재진입 시에만 갱신**되도록 할 수 있다.

### 코드 예시

```typescript
// 찜 해제 시 찜 목록 캐시를 invalidate하지 않음
export const useRemoveFavorite = () => {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: (shopId: number) => del<FavoriteToggleResponse>(`/api/shops/${shopId}/favorite`),
    onSuccess: () => {
      // 찜 목록은 invalidate하지 않음 → 리스트에서 바로 사라지지 않음
      // staleTime: 0이므로 페이지 재진입 시 자동 refetch
      queryClient.invalidateQueries({ queryKey: queryKeys.shops.all });
    },
  });
};

// 찜 목록 조회 — staleTime: 0으로 항상 재진입 시 최신 데이터
export const useFavorites = () => {
  return useQuery({
    queryKey: queryKeys.favorites.all,
    queryFn: () => get("/api/users/me/favorites"),
    staleTime: 0, // 컴포넌트 마운트 시 항상 refetch
  });
};
```

### 핵심 포인트

- `staleTime: 0` → 데이터가 즉시 stale 상태 → 컴포넌트 (재)마운트 시 자동 refetch
- `invalidateQueries` 제거 → 현재 페이지에서 캐시가 갱신되지 않아 아이템 유지
- 하트 UI는 `useFavorite` 훅의 **optimistic update**(useOptimistic)로 즉시 반영

## 3. Capacitor iOS 빌드 파이프라인

### 설명

Capacitor는 웹앱을 네이티브 iOS/Android 앱으로 감싸는 프레임워크다. Next.js의 static export(`output: "export"`)로 생성된 `out/` 디렉토리를 iOS 프로젝트에 동기화(sync)하여 앱을 빌드한다.

### 빌드 플로우

```
npm run build:capacitor     # NEXT_PUBLIC_BUILD_TARGET=capacitor next build
    ↓
npx cap sync ios            # out/ → ios/App/App/public/ 동기화
    ↓
npx cap open ios            # Xcode 열기
    ↓
Xcode: Product → Archive    # 앱 아카이브
    ↓
Distribute App              # App Store Connect 업로드
```

### 핵심 설정

```typescript
// capacitor.config.ts
const config: CapacitorConfig = {
  webDir: "out", // Next.js static export 출력 디렉토리
  server: {
    androidScheme: "https",
  },
};

// next.config.mjs
const nextConfig = {
  ...(isCapacitor && { output: "export" }),
  images: {
    // Capacitor에서는 이미지 최적화 비활성화 필수
    unoptimized: isCapacitor || process.env.NODE_ENV === "development",
  },
};
```

### 주의사항

- Capacitor CLI는 **Node.js >= 22.0.0** 필요 (`nvm use 24` 등으로 전환)
- `next/image`의 `unoptimized: true`가 Capacitor 빌드에서 필수
- App Store 제출 시 **빌드 번호(CURRENT_PROJECT_VERSION)** 중복 불가

## 4. React lint: refs & setState in effect

### 설명

React 19의 strict lint 규칙은 두 가지를 금지한다:

1. **렌더 중 ref.current 접근 금지** (`react-hooks/refs`) — ref는 렌더링에 필요한 값이 아니므로 이벤트 핸들러나 effect에서만 접근해야 함
2. **effect 내 동기적 setState 금지** (`react-hooks/set-state-in-effect`) — 연쇄 렌더링(cascading renders)을 유발

### 잘못된 패턴과 대안

```tsx
// ❌ effect 내 setState
useEffect(() => {
  if (data !== undefined) {
    setSnapshot(data); // lint error: set-state-in-effect
  }
}, [data]);

// ❌ 렌더 중 ref 접근
const ref = useRef(data);
if (data && ref.current === undefined) {
  ref.current = data; // lint error: refs during render
}
const value = ref.current; // lint error

// ✅ 대안: mutation 레벨에서 제어
// 캐시 invalidation을 제거하여 데이터 갱신 자체를 방지
```

### 핵심 포인트

- "스냅샷을 유지한다"는 요구사항을 **컴포넌트 상태로 해결**하려 하면 lint 규칙과 충돌
- **데이터 소스 레벨**(React Query invalidation)에서 해결하는 것이 더 깔끔

## 5. Xcode CURRENT_PROJECT_VERSION (빌드 번호)

### 설명

App Store에 앱을 제출할 때 `Version`(MARKETING_VERSION)과 `Build`(CURRENT_PROJECT_VERSION) 조합이 고유해야 한다. 같은 버전에 같은 빌드 번호로 재제출하면 Apple이 거부한다.

### 변경 방법

`project.pbxproj`에서 Debug/Release 두 곳 모두 수정:

```
// ios/App/App.xcodeproj/project.pbxproj
CURRENT_PROJECT_VERSION = 4;  // 빌드 번호
MARKETING_VERSION = 1.0;      // 버전 번호
```

또는 Xcode GUI: **App 타겟 → General → Build** 값 변경

### 핵심 포인트

- Version(1.0) = 사용자에게 보이는 버전
- Build(4) = 내부 빌드 번호, 같은 Version 내에서 증가
- `project.pbxproj`에 Debug/Release **두 곳** 모두 존재하므로 둘 다 변경 필요

- A) `NSAllowsArbitraryLoads`
- B) `NSAllowsArbitraryLoadsInWebContent`
- C) `NSExceptionAllowsInsecureHTTPLoads`

<details>
<summary>정답 보기</summary>

**B) NSAllowsArbitraryLoadsInWebContent**

`NSAllowsArbitraryLoads`는 앱 전체에 적용되어 Apple 심사에서 거부될 수 있다. `NSAllowsArbitraryLoadsInWebContent`는 WebView 내부에만 적용되므로 Capacitor 앱에 적합하다.

</details>

**Q2. React Query에서 `staleTime: 0`으로 설정하면 어떤 동작이 발생할까?**

- A) 데이터를 절대 캐싱하지 않는다
- B) 쿼리를 관찰하는 컴포넌트가 마운트될 때마다 refetch한다
- C) 데이터 변경 시 자동으로 mutation을 실행한다
- D) 서버에 매 초마다 polling한다

<details>
<summary>정답 보기</summary>

**B) 쿼리를 관찰하는 컴포넌트가 마운트될 때마다 refetch한다**

`staleTime: 0`은 데이터를 즉시 stale 상태로 만든다. 캐시 자체는 유지되지만(cacheTime이 별도), 컴포넌트가 마운트되어 쿼리를 관찰하면 stale 데이터이므로 백그라운드 refetch를 트리거한다.

</details>

**Q4. Capacitor에서 Next.js 앱을 iOS로 빌드할 때 `images.unoptimized: true`가 필요한 이유는?**

- A) iOS에서 WebP 형식을 지원하지 않아서
- B) Next.js Image Optimization API 서버가 없는 static export이므로
- C) Capacitor가 이미지 캐싱을 자체적으로 처리해서
- D) App Store 심사 가이드라인에서 요구해서

<details>
<summary>정답 보기</summary>

**B) Next.js Image Optimization API 서버가 없는 static export이므로**

`next/image`는 기본적으로 Next.js 서버의 Image Optimization API를 통해 이미지를 처리한다. Capacitor 빌드는 `output: "export"`로 정적 파일만 생성하므로 서버가 없다. `unoptimized: true`로 설정해야 이미지가 원본 그대로 `<img>` 태그로 렌더링된다.

</details>

**Q5. React 19 strict lint에서 `useEffect` 내부에서 `setState`를 직접 호출하면 어떤 문제가 발생한다고 경고할까?**

- A) 메모리 누수
- B) 연쇄 렌더링 (cascading renders)
- C) 무한 루프
- D) 서버-클라이언트 하이드레이션 불일치

<details>
<summary>정답 보기</summary>

**B) 연쇄 렌더링 (cascading renders)**

Effect 내에서 동기적으로 setState를 호출하면 불필요한 추가 렌더가 발생한다. React는 effect를 외부 시스템과의 동기화 용도로 사용하길 권장하며, 상태 변환은 렌더 중이나 이벤트 핸들러에서 처리하는 것이 적합하다.

</details>
