## 버그 컨텍스트

ShopPreviewBottomSheet에서 바텀시트가 풀 스크린으로 확장된 상태에서 상단 액션 버튼(찜, 공유, 메뉴)을 클릭하면 바텀시트가 닫히는 버그를 수정하면서 이벤트 버블링과 브라우저 이벤트 순서에 대해 학습했다.

---

## 배운 개념

### 1. 이벤트 버블링 (Event Bubbling)

DOM에서 이벤트가 발생하면 해당 요소에서 시작하여 부모 → 조부모 → ... → document까지 올라가는 전파 방식.

```
<div onMouseDown={드래그시작}>        ← 부모
  <button onClick={찜하기}>           ← 자식 (이벤트 발생)
</div>

// 버튼 클릭 시 → mousedown이 부모까지 버블링 → 드래그 로직 실행
```

- **대부분의 이벤트**가 버블링됨 (click, mousedown, touchstart 등)
- **버블링 안 되는 예외**: `focus`, `blur`, `mouseenter`, `mouseleave`, `scroll`

### 2. 이벤트 전파 3단계

브라우저에서 이벤트는 3단계를 거친다:

```
     document
        ↓  ① 캡처링 (위 → 아래)
      <div>
        ↓
    <button>   ② 타겟 (이벤트 발생 지점)
        ↑
      <div>
        ↑  ③ 버블링 (아래 → 위)
     document
```

- 캡처링이 먼저, 버블링이 나중
- 일반적으로 핸들러는 버블링 단계에서 실행됨

### 3. 브라우저 마우스 이벤트 발생 순서

하나의 클릭에서 실제로는 여러 이벤트가 순서대로 발생한다:

```
1. mousedown   ← 손가락이 화면에 닿는 순간
2. mouseup     ← 손가락을 떼는 순간
3. click       ← mousedown + mouseup이 완료된 후 발생
```

`click`은 `mousedown` + `mouseup`이 완료되어야 만들어지는 **복합 이벤트**이므로 항상 가장 마지막에 발생한다.

### 4. stopPropagation vs preventDefault

```js
e.stopPropagation(); // 부모로 전파(버블링)를 막음
e.preventDefault(); // 브라우저 기본 동작을 막음 (폼 제출, 링크 이동 등)
```

- 이번 버그에서는 자식 버튼의 이벤트가 부모 드래그 핸들러로 전파되는 것을 막아야 했으므로 `stopPropagation` 사용

### 5. e.target vs e.currentTarget

```jsx
<div
  onClick={(e) => {
    e.target; // 실제 클릭된 요소 (자식 button일 수 있음)
    e.currentTarget; // 핸들러가 붙은 요소 (항상 이 div)
  }}
>
  <button>클릭</button>
</div>
```

버블링으로 인해 `target`과 `currentTarget`이 다를 수 있음.

### 6. 이벤트 위임 (Event Delegation)

버블링을 활용하여 부모 요소 하나에만 핸들러를 등록하는 패턴:

```jsx
// 부모 하나에 핸들러 → 메모리 효율적
<ul onClick={(e) => {
  const id = (e.target as HTMLElement).dataset.id;
  if (id) handle(id);
}}>
  {items.map(item => <li data-id={item.id}>...</li>)}
</ul>
```

React도 내부적으로 이 패턴을 사용한다.

### 7. useRef 잔여값 문제

useRef의 `.current` 값은 렌더링과 무관하게 유지되므로, 이전 인터랙션의 값이 남아있을 수 있다.

```js
// 이전 드래그에서 lastDragY.current = 200이 남아있으면
// 새 클릭 시 잘못된 거리 계산 발생
const handleDragStart = (clientY: number) => {
  dragStartY.current = clientY;
  lastDragY.current = clientY;  // ← 반드시 초기화 필요
};
```

## 버그 수정 요약

| 수정                               | 내용                       | 목적                                        |
| ---------------------------------- | -------------------------- | ------------------------------------------- |
| `lastDragY.current = clientY` 추가 | handleDragStart에서 초기화 | 잔여값으로 인한 오판 방지 (방어적 코딩)     |
| `e.stopPropagation()` 추가         | 액션 버튼 컨테이너에 적용  | 버튼 클릭이 드래그를 트리거하지 않도록 차단 |

## 퀴즈 (5/5 정답)

**Q1. 다음 중 버블링이 발생하지 않는 이벤트는?**

- A) click
- B) mousedown
- C) focus
- D) touchstart

<details>
<summary>정답 보기</summary>

C) focus ✅

focus, blur, mouseenter, mouseleave, scroll은 버블링되지 않는 예외 이벤트

</details>

**Q2. 버튼을 한 번 클릭했을 때 브라우저 이벤트 발생 순서로 올바른 것은?**

- A) click → mousedown → mouseup
- B) mousedown → click → mouseup
- C) mousedown → mouseup → click
- D) mouseup → mousedown → click

<details>
<summary>정답 보기</summary>

C) mousedown → mouseup → click ✅

click은 mousedown + mouseup이 완료되어야 생성되는 복합 이벤트이므로 항상 마지막

</details>

**Q3. stopPropagation과 preventDefault에 대한 설명으로 올바른 것은?**

- A) 둘 다 부모로의 이벤트 전파를 막는다
- B) stopPropagation은 전파를 막고, preventDefault는 브라우저 기본 동작을 막는다
- C) preventDefault는 전파를 막고, stopPropagation은 브라우저 기본 동작을 막는다
- D) 둘 다 브라우저 기본 동작을 막는다

<details>
<summary>정답 보기</summary>

B) stopPropagation은 전파를 막고, preventDefault는 브라우저 기본 동작을 막는다 ✅

서로 완전히 다른 역할. 이번 버그에서는 전파 차단이 필요했으므로 stopPropagation 사용

</details>

**Q4. O/X: 이벤트 전파에서 캡처링 단계가 버블링 단계보다 먼저 실행된다.**

<details>
<summary>정답 보기</summary>

O ✅

캡처링(위→아래) → 타겟 → 버블링(아래→위) 순서

</details>

**Q5. 버튼을 클릭했을 때 부모 div의 onClick에서 e.target과 e.currentTarget은?**

```jsx
<div onClick={(e) => console.log(e.target, e.currentTarget)}>
  <button>클릭</button>
</div>
```

- A) 둘 다 div
- B) 둘 다 button
- C) target: button, currentTarget: div
- D) target: div, currentTarget: button

<details>
<summary>정답 보기</summary>

C) target: button, currentTarget: div ✅

target은 실제 클릭된 요소(button), currentTarget은 핸들러가 등록된 요소(div)

</details>
