### 1. Capacitor dev:ios - iOS 실기기 테스트 워크플로우

Capacitor 기반 Next.js 앱을 iPhone에서 테스트하는 두 가지 방식:

**개발 모드 (라이브 서버 연결)**

```bash
npm run dev:ios
# CAP_DEV_SERVER=true → capacitor.config.ts의 server.url을 사용
# 로컬 변경사항이 반영되지 않음 (dev 서버 URL을 바라봄)
```

**정적 빌드 모드 (로컬 파일)**

```bash
npm run build:capacitor  # NEXT_PUBLIC_BUILD_TARGET=capacitor next build
npm run sync:ios          # npx cap sync ios (out/ → ios/App/App/public/)
# Xcode에서 빌드 & Run
```

핵심: `dev:ios`는 외부 dev 서버를 바라보므로 로컬 코드 변경을 테스트하려면 **빌드 후 sync**가 필요.

### 2. CSS dvh vs vh - Capacitor iOS WebView 호환성

| 단위  | 설명                                | Capacitor 지원   |
| ----- | ----------------------------------- | ---------------- |
| `vh`  | 고정 뷰포트 높이 (주소창 무시)      | 안정적           |
| `dvh` | 동적 뷰포트 높이 (주소창 높이 반영) | 불안정할 수 있음 |

**문제**: `h-[calc(100dvh-env(safe-area-inset-top,0px))]` 사용 시, Capacitor iOS WebView에서 `dvh`가 제대로 동작하지 않으면 컨테이너 높이가 무효화되어 `overflow-hidden`이 작동하지 않고 헤더가 스크롤됨.

**해결**: CSS fallback 패턴 활용 - 같은 속성을 두 번 선언하면 브라우저가 마지막 유효한 값을 사용.

```css
/* globals.css */
@layer utilities {
  .h-safe-viewport {
    height: calc(100vh - env(safe-area-inset-top, 0px)); /* fallback */
    height: calc(100dvh - env(safe-area-inset-top, 0px)); /* modern */
  }
}
```

CSS 원리: 브라우저는 같은 속성이 두 번 선언되면 **마지막 유효한 값**을 사용. `dvh` 미지원 시 두 번째 줄이 무시되고 첫 번째 `vh` 값이 적용됨.

### 3. Next.js Static Export + Dynamic Routes

`output: "export"` 모드에서 동적 라우트(`/shop/[id]`)의 동작 방식:

**웹 (Next.js 서버)**

```
router.push('/shop/123')
→ 서버가 동적으로 페이지 렌더링
→ 성공
```

**앱 (Capacitor 정적 파일)**

```
router.push('/shop/123')
→ 로컬에서 /shop/123.html 파일 탐색
→ generateStaticParams()에서 [{ id: "0" }]만 생성
→ /shop/0.html만 존재
→ 파일 없음 → 실패
```

**해결법**: Capacitor에서는 pre-rendered 페이지로 이동하면서 query param으로 실제 ID 전달.

```tsx
// 네비게이션 코드
import { isNativeApp } from "@/utils/platform";

const handleShopClick = (shopId: number) => {
  if (isNativeApp()) {
    router.push(`/shop/0?shopId=${shopId}`); // pre-rendered 페이지 + query param
  } else {
    router.push(`/shop/${shopId}`); // 서버가 동적 처리
  }
};

// ShopDetailClient.tsx - query param 우선 읽기
const searchParams = useSearchParams();
const params = useParams();
const shopId = parseShopId(searchParams.get("shopId") ?? params.id);
```

### 4. useSearchParams + Suspense 경계

Next.js 정적 빌드(`output: "export"`)에서 `useSearchParams()`를 사용하면 **반드시 Suspense 경계**가 필요.

**이유**: 서버 사이드 렌더링(SSR/SSG) 시점에는 URL의 query parameter 값을 알 수 없음. `useSearchParams()`는 클라이언트에서만 값을 읽을 수 있으므로 Next.js가 Suspense로 감싸서 클라이언트 렌더링으로 폴백하도록 요구.

**에러 메시지**:

```
useSearchParams() should be wrapped in a suspense boundary at page "/shop/[id]"
```

**해결**:

```tsx
// page.tsx (Server Component)
import { Suspense } from "react";
import ShopDetailClient from "./ShopDetailClient";

export default function ShopDetailPage() {
  return (
    <Suspense>
      <ShopDetailClient />
    </Suspense>
  );
}
```

- 참고: https://nextjs.org/docs/messages/missing-suspense-with-csr-bailout
