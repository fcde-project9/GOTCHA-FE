## 1. Capacitor `server.url`의 역할

WebView가 **로드할 웹페이지 URL**을 지정하는 설정. API 주소가 아님.

| 설정              | WebView 동작                                 |
| ----------------- | -------------------------------------------- |
| `server.url` 없음 | `webDir`(`out/`)의 로컬 정적 파일을 로드     |
| `server.url` 있음 | 로컬 파일 무시, 해당 원격 URL을 WebView 로드 |

- **개발 시**: `server.url`에 dev 서버 주소를 넣어 빌드 없이 빠른 테스트
- **배포 시**: `server.url`을 **반드시 제거**하여 로컬 번들 사용

```ts
// capacitor.config.ts
server: {
  // url: "https://dev.gotcha.it.com", // 개발 전용. 배포 시 제거
  androidScheme: "https",
},
```

## 2. 커스텀 URL 스킴의 `new URL()` 파싱 동작

`gotchaapp://oauth/callback?code=abc`를 `new URL()`로 파싱하면 HTTP URL과 다르게 분해됨.

| 속성           | 값             |
| -------------- | -------------- |
| `url.protocol` | `"gotchaapp:"` |
| `url.hostname` | `"oauth"`      |
| `url.pathname` | `"/callback"`  |

`url.pathname`은 항상 truthy이므로 `url.pathname || url.hostname + url.pathname` 패턴은 항상 `pathname`만 반환 → `"oauth/callback"` 매칭 실패.

```ts
// 수정: 커스텀 스킴 여부에 따라 경로 조합 방식 분기
const isCustomScheme = url.protocol !== "http:" && url.protocol !== "https:";
const path = isCustomScheme ? url.hostname + url.pathname : url.pathname;
// "oauth" + "/callback" = "oauth/callback" ✓
```

## 3. PluginListenerHandle을 이용한 리스너 정리

Capacitor 플러그인의 `addListener()`는 `PluginListenerHandle`을 반환. `removeAllListeners()` 대신 개별 핸들의 `remove()`를 호출해야 다른 리스너에 영향을 주지 않음.

### Race condition 방지 패턴

React `useEffect` cleanup이 `addListener` Promise resolve보다 먼저 실행될 수 있음. `cancelled` 플래그로 방어.

```ts
let handle: Awaited<ReturnType<typeof App.addListener>> | undefined;
let cancelled = false;

App.addListener("appUrlOpen", handleUrlOpen).then((h) => {
  if (cancelled) {
    h.remove(); // cleanup이 먼저 실행된 경우 즉시 제거
  } else {
    handle = h;
  }
});

return () => {
  cancelled = true;
  handle?.remove();
};
```

## 4. APNs 환경 설정 (`aps-environment`)

`App.entitlements`의 `aps-environment` 값이 APNs 연결 환경을 결정.

| 설정값        | 용도                   |
| ------------- | ---------------------- |
| `development` | sandbox APNs (개발)    |
| `production`  | production APNs (배포) |

**Xcode 자동 서명(Automatic Signing) 사용 시**, `production`으로 설정해도 Debug 빌드에서는 Xcode가 프로비저닝 프로파일에 맞게 sandbox APNs를 사용하므로 개발 푸시 테스트에 영향 없음. `development`로 두면 TestFlight/App Store 빌드에서 프로덕션 APNs에 연결 불가.

> **주의 (수동 서명):** Manual Signing을 사용하면 Xcode가 `.entitlements` 값을 빌드 설정별로 덮어쓰지 않으므로, `production`이 하드코딩된 상태에서 개발 빌드를 하면 production APNs에 연결되어 디바이스 토큰이 유효하지 않을 수 있음. 수동 서명 시에는 Debug/Release별로 별도 entitlements 파일을 사용하거나 빌드 설정에서 분기 필요.

## 5. AppDelegate APNs 콜백 메서드

`@capacitor/push-notifications` 플러그인이 APNS 토큰 등록 결과를 수신하려면 AppDelegate에 두 콜백 메서드가 필수.

```swift
func application(_ application: UIApplication,
                 didRegisterForRemoteNotificationsWithDeviceToken deviceToken: Data) {
    NotificationCenter.default.post(
        name: .capacitorDidRegisterForRemoteNotifications, object: deviceToken)
}

func application(_ application: UIApplication,
                 didFailToRegisterForRemoteNotificationsWithError error: Error) {
    NotificationCenter.default.post(
        name: .capacitorDidFailToRegisterForRemoteNotifications, object: error)
}
```

없으면 `registration` / `registrationError` 이벤트가 발생하지 않아 토큰을 받을 수 없음.

## 6. SplashScreen 안전한 숨기기 패턴

`SplashScreen.hide()`가 **어떤 상황에서도** 호출되도록 보장하는 패턴.

### 핵심: dynamic import도 try 안에 넣기

```ts
const initNativeApp = async () => {
  try {
    const { StatusBar, Style } = await import("@capacitor/status-bar");
    const { Keyboard, KeyboardResize } = await import("@capacitor/keyboard");
    await StatusBar.setStyle({ style: Style.Light });
    await Keyboard.setResizeMode({ mode: KeyboardResize.Body });
  } finally {
    try {
      const { SplashScreen } = await import("@capacitor/splash-screen");
      await SplashScreen.hide();
    } catch {
      // SplashScreen import 자체가 실패하면 무시
    }
  }
};
```

### 왜 SplashScreen을 수동으로 숨겨야 하는가?

일반 네이티브 앱(Swift/Kotlin)은 LaunchScreen이 끝나면 네이티브 UI가 즉시 준비되어 있어 별도 처리가 불필요하다. 하지만 Capacitor 같은 **하이브리드 앱**은 WebView가 HTML/JS/CSS를 로드하는 시간이 필요하므로, LaunchScreen이 끝나도 **빈 흰 화면이 노출**되는 문제가 생긴다.

이를 방지하기 위해 Capacitor는 **자체 스플래시 레이어를 WebView 위에 덮어두고**, 웹앱이 준비되면 `SplashScreen.hide()`로 걷는 구조를 사용한다.

```text
일반 앱:  앱 실행 → LaunchScreen → 네이티브 UI 즉시 렌더링
Capacitor: 앱 실행 → LaunchScreen → 스플래시(WebView 가림) → JS 로드 → hide() → 웹앱 노출
```

### 설정 포인트

- `launchAutoHide: false`로 설정하여 수동 제어 (자동 숨기면 초기화 완료 전에 흰 화면 노출 가능)
- import 실패, API 호출 실패 모두 `finally`에서 `hide()` 호출하여 앱이 멈춰 보이지 않도록 보장

## 7. Capacitor 플러그인 dynamic import 패턴

네이티브 전용 플러그인은 top-level static import 대신 **사용 시점에 dynamic import**하는 것이 권장.

```ts
// Bad: 웹에서도 모듈이 로드됨
import { PushNotifications } from "@capacitor/push-notifications";

// Good: 네이티브 분기에서만 로드
async function registerNativePush() {
  const { PushNotifications } = await import("@capacitor/push-notifications");
  // ...
}
```

이유: 웹 빌드에서 불필요한 네이티브 모듈 로드 방지, 트리셰이킹 개선.

## 8. 일회성 리스너 패턴 (리스너 누적 방지)

`addListener`가 반복 호출될 때 리스너가 누적되는 문제를 핸들의 `remove()`로 해결.

```ts
// 토큰 수신 후 즉시 리스너 제거
const regHandle = await PushNotifications.addListener("registration", async (token) => {
  await regHandle.remove(); // 일회성: 수신 후 즉시 제거
  await apiClient.post(ENDPOINTS.PUSH.REGISTER_DEVICE, {
    token: token.value,
    platform: getPlatform(), // "ios" 하드코딩 대신 동적 감지
  });
});
```
