## 주제: React Query Optimistic Update와 캐시 무효화 전략

### 1. Optimistic Update 후 `onSettled` invalidation의 함정

**문제**: `onSettled`에서 `invalidateQueries`를 호출하면, 성공 시에도 즉시 refetch가 트리거된다. 백엔드의 read model이 아직 반영되지 않은 상태라면, **stale 데이터가 optimistic update 값을 덮어쓴다.**

**흐름 (버그):**

```
1. onMutate → optimistic update (likeCount: 0 → 1) ✅
2. API 성공 → 서버 응답 { isLiked: true }
3. onSettled → invalidateQueries → refetch → 서버 아직 반영 안 됨 → likeCount: 0 ❌
```

**해결**: 성공 시에는 `invalidateQueries`를 호출하지 않고 optimistic update 값을 유지. 실패 시에만 롤백 + invalidation.

```ts
// 기존 (버그)
onSettled: (_, __, { shopId }) => {
  // 성공/실패 모두 refetch → stale 데이터로 덮어쓰기 가능
  queryClient.invalidateQueries({ queryKey: queryKeys.reviews.byShop(shopId) });
  queryClient.invalidateQueries({ queryKey: ["shops", "detail", shopId] });
},

// 수정 후
onError: (_, { shopId }, context) => {
  // 롤백
  context?.previousDetail?.forEach(([queryKey, data]) => {
    queryClient.setQueryData(queryKey, data);
  });
  // 실패 시에만 서버 동기화
  queryClient.invalidateQueries({ queryKey: queryKeys.reviews.byShop(shopId) });
  queryClient.invalidateQueries({ queryKey: ["shops", "detail", shopId] });
},
// onSettled 제거 → 성공 시 optimistic update 값 유지
```

### 2. API 응답 타입과 실제 응답 불일치 주의

**문제**: `ReviewLikeResponse` 타입에 `likeCount`가 없는데, `onSuccess`에서 `data.likeCount`를 사용하면 `undefined`가 캐시에 세팅된다.

```ts
// 타입 정의 (API 스펙 기준)
interface ReviewLikeResponse {
  reviewId: number;
  isLiked: boolean;
  // likeCount 없음!
}

// onSuccess에서 사용 시
const { isLiked, likeCount } = data; // likeCount = undefined
// → review.likeCount가 undefined로 세팅 → 화면에 숫자 안 보임
```

**교훈**: API 응답 타입은 실제 API 스펙과 정확히 일치해야 하며, 타입에 없는 필드를 사용하면 런타임에 `undefined`가 되어 예상치 못한 버그가 발생한다.

### 3. Mutation 후 관련 캐시 모두 invalidate하기

**문제**: 리뷰 수정 후 `reviews.byShop(shopId)` 캐시만 invalidate하고, `shops.detail(shopId)` 캐시는 invalidate하지 않아서 shop detail의 미리보기 리뷰가 갱신되지 않았다.

```ts
// 기존 (리뷰 캐시만 무효화)
onSuccess: () => {
  queryClient.invalidateQueries({ queryKey: queryKeys.reviews.byShop(shopId) });
},

// 수정 후 (shop detail 캐시도 무효화)
onSuccess: () => {
  queryClient.invalidateQueries({ queryKey: queryKeys.reviews.byShop(shopId) });
  queryClient.invalidateQueries({ queryKey: ["shops", "detail", shopId] });
},
```

**교훈**: 하나의 엔티티(리뷰)가 여러 쿼리 캐시에 존재할 수 있다. mutation 후에는 해당 데이터를 포함하는 **모든** 관련 캐시를 invalidate해야 한다.

## 퀴즈

**Q1. React Query의 `onSettled` 콜백은 언제 실행되나요?**

- A) mutation 성공 시에만
- B) mutation 실패 시에만
- C) mutation 성공/실패 모두
- D) mutation 시작 시

<details>
<summary>정답 보기</summary>
C) mutation 성공/실패 모두

`onSettled`는 mutation이 완료되면 결과에 관계없이 항상 실행됩니다. 따라서 여기서 `invalidateQueries`를 호출하면 성공 시에도 refetch가 발생합니다.

</details>

**Q2. Optimistic update 후 `onSettled`에서 `invalidateQueries`를 호출하면, 백엔드 반영이 늦을 때 어떤 문제가 발생하나요?**

- A) 네트워크 에러가 발생한다
- B) Optimistic update 값이 stale 데이터로 덮어써진다
- C) 캐시가 완전히 삭제된다
- D) 컴포넌트가 unmount된다

<details>
<summary>정답 보기</summary>
B) Optimistic update 값이 stale 데이터로 덮어써진다

`invalidateQueries`가 트리거하는 refetch가 아직 반영되지 않은 서버 데이터를 가져와서, 올바른 optimistic update 값을 덮어씁니다.

</details>

**Q3. O/X: TypeScript에서 `{ id: number; name: string }` 타입의 객체를 spread하면, 결과 타입에 원본 객체의 다른 필드(예: `email`)도 포함된다.**

<details>
<summary>정답 보기</summary>
X

TypeScript는 선언된 타입 기준으로 추론합니다. 파라미터가 `{ id: number; name: string }`으로 선언되면, 런타임에 다른 필드가 있더라도 타입 시스템에서는 `id`와 `name`만 인식합니다. spread 결과도 좁은 타입으로 추론됩니다.

</details>

**Q4. 리뷰 데이터가 `ShopDetailResponse.reviews`와 `useInfiniteReviews` 두 곳에 캐시되어 있을 때, 리뷰 수정 mutation의 `onSuccess`에서 해야 할 것은?**

- A) 하나의 캐시만 invalidate하면 된다
- B) 두 캐시 모두 invalidate해야 한다
- C) 캐시 invalidation은 필요 없다
- D) 페이지를 강제 새로고침한다

<details>
<summary>정답 보기</summary>
B) 두 캐시 모두 invalidate해야 한다

하나의 엔티티가 여러 쿼리 캐시에 존재할 수 있으므로, mutation 후에는 해당 데이터를 포함하는 모든 관련 캐시를 invalidate해야 UI가 일관되게 갱신됩니다.

</details>
